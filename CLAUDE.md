# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A static, no-build, no-dependency site for publishing personal game reviews, plus a single-file admin tool for authoring them. There is no `package.json`, no bundler, and no test suite — it's six flat files (`index.html`, `app.js`, `admin.html`, `shared.js`, `style.css`, `serve.ps1`) served as-is.

## Running it locally

```powershell
powershell -ExecutionPolicy Bypass -File .\serve.ps1        # serves the repo root at http://localhost:8080 (default port 8080)
powershell -ExecutionPolicy Bypass -File .\serve.ps1 -Port 3000
```

`serve.ps1` is a minimal `System.Net.HttpListener`-based static file server (no Node/npm involved). There is no build, lint, or test command — edit the HTML/JS/CSS files directly and reload the browser.

## Architecture

Two front ends share one data model:

- **`index.html` + `app.js`** — the public site. Renders review cards, tag filtering, expandable card detail with nested review tabs/sub-tabs.
- **`admin.html`** — a single file with all its CSS and JS inline (no separate admin.js). Used to author/edit/delete reviews and push them to GitHub, plus preview exactly how a review will look both as a site card and as a Steam review.
- **`shared.js`** — constants and helpers used identically by both of the above (`escHtml`, `formatDate`, `steamAppId`, `splitSubTabs`, `renderBBCode`, `renderCard`, `STEAM_ICON`, `SITE_LINK_TEXT`). **Anything both files need must live here, not be duplicated in each** — this file used to not exist and the same logic/text drifted out of sync between the two UIs. `admin.html` loads it via a normal blocking `<script src="shared.js">` in `<head>`; `index.html` loads it dynamically alongside `app.js` (see below) with `.async = false` on both loader tags, in that order, so `shared.js` is guaranteed to finish before `app.js` runs even though both are non-parser-inserted (and thus otherwise unordered) script tags.

### No native browser dialogs in the admin UI

`admin.html` uses no `prompt()`/`alert()`/`confirm()` anywhere — build inline UI instead:
- **Text input** (naming/entering something): swap the trigger button for an `<input>` with its own confirm/cancel controls, focused (and often pre-filled). Most of these are built on the shared `startInlineEditor(triggerId, editorId, inputId, opts)`/`cancelInlineEditor(triggerId, editorId)`/`onInlineEditorKeydown(e, onConfirm, onCancel)` helpers (`shared` here meaning defined once in `admin.html` and reused, not `shared.js` — this trio is admin-only UI plumbing, not something the public site needs) — **use these for any new trigger/editor flow instead of hand-rolling the show/hide/focus/keydown wiring again.** `opts.value` pre-fills the input (e.g. renaming), `opts.select` also selects it; pass `inputId: null` for an editor with no input (e.g. `startDeleteReview`). Current callers: `startAddTagCategory`, `startInsertLink`, `startSaveCustomBlock`, `startSaveCustomTab`, `startAddTag`, `startDeleteReview`, and the tag-category rename flow (`startRenameTagCategory`/`confirmRenameTagCategory`, which still re-renders its own row via `renderTags()` since — unlike the others — more than one can exist at once, but still uses `onInlineEditorKeydown` for its Enter/Escape handling). The one holdout is the tag input's own `onTagKeydown`, which keeps a custom handler because it does more than confirm/cancel (arrow-key suggestion navigation, comma-to-add, backspace-to-remove-last-chip).
- **Destructive confirmation**: replace the row/button with an inline "Delete X? ✓ ✕" state instead of `confirm()`. The label + ✓/✗ pair itself is `deleteConfirmRow(label, onConfirmCall, onCancelCall)` (returns HTML; `onConfirmCall`/`onCancelCall` are JS-expression strings, e.g. `` `confirmDeleteCustomBlock('${b.id}')` ``) — callers wrap it in whatever container fits their context (a `.dropdown-item.dropdown-item-confirm` row for saved-block/saved-tab delete; a bare `<span>` for the inline tag-category delete). The standalone review Delete button (`startDeleteReview`/`confirmDeleteReview`/`cancelDeleteReview`, using the `startInlineEditor`/`cancelInlineEditor` helpers above) is the one exception with its own text/labels ("Yes, delete"/"Cancel") rather than `deleteConfirmRow`'s ✓/✗ icons, since it predates that shared shape and reads better spelled out for a destructive whole-review action.
- **Errors**: go through `#error-banner` (`showError`/`setSt(id, msg, true)`), never `alert()`.

Keep new interactive admin features on this pattern — reach for `startInlineEditor`/`cancelInlineEditor`/`onInlineEditorKeydown`/`deleteConfirmRow` first, and only hand-roll a bespoke `start*/confirm*/cancel*` trio when a flow's mechanics genuinely don't fit (extra key handling, no fixed trigger/editor DOM pair, etc.) rather than by default.

### Data model & persistence

A review is a plain object: `{ id, title, recommended, summary, coverImage, hoursPlayed, datePosted, body, tabs: [{ id, name, content }], tags: [] }`. `body` (and each tab's `content`) is BBCode-ish markup (see below), not HTML.

Reviews are stored as a single JSON array at `data/reviews.json` inside a **GitHub repo the user configures in the admin UI** (Settings → owner/repo/branch/PAT, saved to `localStorage`). `admin.html` reads/writes that file directly through GitHub's REST Contents API, via the generic `ghGetJson(path)`/`ghPutJson(path, data, sha, msg)` helpers (which wrap `ghReq` and base64-encode/decode the content — `b64encode`/`b64decode`, UTF-8 safe). `fetchFile`/`pushFile` are thin wrappers of those two around `data/reviews.json`; `fetchBlocksFile`/`pushBlocksFile` are the same for `data/custom-blocks.json` (see below). Any new GitHub-persisted file that's saved on its own (not bundled into another commit, see the Git Data API note below) should go through `ghGetJson`/`ghPutJson` rather than re-implementing the Contents API calls.

Every write `ghPutJson` makes is authored as a fixed `DATA_COMMIT_IDENTITY` (`Review Admin <review-admin@localhost>`), not the PAT owner — deliberately not a verified email on any GitHub account, so these automated data-saving commits don't show up on the token owner's contribution graph. The same identity is used for the Git Data API commits below.

`saveReview()` normally just calls `pushFile` (single-file Contents API write, like blocks/tabs). But if there are pending tag/category edits (`tagsDirty`, see the Tag categories section below), it instead calls `ghCommitFiles([{path, data}, ...], message)` — a small Git Data API layer (`ghGetRef`/`ghGetCommitTree`/`ghCreateBlob`/`ghCreateTree`/`ghCreateCommit`/`ghUpdateRef`) that commits `data/reviews.json`, `data/tag-categories.json`, and `data/tags.json` together as **one** atomic commit, instead of three separate Contents-API pushes. Reach for `ghCommitFiles` (not three `ghPutJson` calls) any time a save needs to span more than one file in a single commit.

**Important:** the public site (`app.js`) never calls the *authenticated* GitHub API. `init()` reads the `gh_reviews_cache`/`gh_tags_cache`/`gh_tagcats_cache` `localStorage` keys, which `admin.html` populates whenever it successfully loads or saves reviews (`cacheReviews()`/`cacheTagCategories()`/`cacheTagRegistry()`). So a browser where the admin tool has been configured always shows whatever it last loaded/saved there — there's no server-side or build-time fetch of `data/reviews.json` for that path.

A browser that has never had admin configured (`gh_reviews_cache` key absent — an *empty* cached array, e.g. a real repo with zero reviews, is left alone and does **not** trigger this) falls back to `DEFAULT_DATA_REPO` (`shared.js`, currently `jrthayer/ReviewRepo_Data` @ `main`) — a public, read-only default. `fetchDefaultRepoJson(path)` fetches `data/<file>.json` from that repo unauthenticated via `raw.githubusercontent.com` (no base64 decoding needed, unlike the Contents API), resolving to `null` on any failure rather than throwing, since every caller treats "nothing there" as an empty state. `app.js`'s `init()` uses this to show real reviews instead of "No reviews yet." to a fresh visitor; `admin.html`'s `loadDefaultReviewList()` uses the same helper so the review list/editor/tag chips aren't empty either when Settings isn't configured yet, letting you browse and experiment with the form. Neither path writes the fallback data into the `gh_*_cache` keys — those specifically mean "what admin last loaded/saved from **its configured** repo," and conflating the two would make configuring a real repo later behave differently than it should. `loadDefaultReviewList()` also re-checks `getCfg()` before applying its fetched data, in case Settings got configured (and `loadReviewList()` ran) while its fetch was still in flight, so it can't clobber real data with a late-arriving fallback response. Save/Delete already refuse to run without a configured owner/repo/token (`saveReview()`/`startDeleteReview()`), and that guard is what actually prevents writes back to `DEFAULT_DATA_REPO` — `loadDefaultReviewList()` deliberately never touches Settings/`getCfg()`, so that guard keeps firing correctly against fallback data.

### BBCode-style markup (`renderBBCode` in `shared.js`)

Review body text uses a Steam-review-compatible tag syntax: `[b] [i] [u] [strike] [h1] [h2] [h3] [quote] [code] [spoiler] [list]/[olist] with [*] items [table]/[tr]/[th]/[td] [url=...] [hr] [noparse]`. This is deliberate — the same markup a user types in the admin editor is what gets copied verbatim into an actual Steam review (`copyForSteam()`), and is also rendered to HTML for the site card and admin previews. Don't "upgrade" this to Markdown or HTML without accounting for both uses.

A separate, non-BBCode `[tab=Name]...[/tab]` syntax (`splitSubTabs()`) lets a single review tab's body be split into nested sub-tabs on the site (`.sub-tabs`/`.sub-tab`), independent of the top-level review tabs described below.

### Review tabs vs. sub-tabs

Two distinct, similarly-named tab systems:
- **Review tabs** (`.review-tabs`/`.review-tab`, `extraTabs` in admin) — top-level, hang off the top-right corner of a card (see `style.css`'s "book-index tabs" comment), one of which is always the built-in "Steam Review" tab (`id: 'steam'`) backed by `body`; the rest come from the review's `tabs` array and are reorderable by drag-and-drop in the admin UI (`reorderReviewTabs`).
- **Sub-tabs** (`.sub-tabs`/`.sub-tab`) — parsed out of a single tab's content via the `[tab=Name]` markup above, rendered as a small pill row above that tab's body.

### Admin preview mirrors the live site

`admin.html`'s "Site Card Preview" panel (`renderSitePreview`) and the live site's card list (`render` in `app.js`) both build their card HTML from the single shared `renderCard(r, opts)` in `shared.js` — they no longer hand-duplicate the markup. Only the event-wiring on top differs (`app.js` wires up a list of cards sharing one global expand/tab state; `admin.html` wires up a single card that re-renders from the live form on every keystroke), so that part stays local to each file. `renderCard`'s `permalinkHref`/`disablePermalinkNav` options exist because the permalink genuinely needs different behavior per caller: the site uses a same-page relative `?review=` link, the admin preview uses an absolute `siteReviewUrl()` with navigation suppressed (it's a preview, not a real page). `#site-preview` reuses `style.css`'s actual `.card` rules unmodified by re-scoping the same CSS custom property names (`--bg`, `--surface`, `--accent`, etc.), since the admin UI's own dark theme uses different values for those variables.

### Permalinks between the site and Steam

`?review=<id>` on `index.html` auto-expands and scrolls to that card on load (see `init()` in `app.js`). `admin.html`'s `siteReviewUrl(id)` builds that URL, and it's appended (using the shared `SITE_LINK_TEXT` copy) both to the site card itself (`.card-permalink`) and to the Steam-review text/preview — so a posted Steam review links back to the fuller write-up on the site, and the site card links to itself for sharing.

### Custom format blocks & saved tabs

Two parallel "save this for reuse" features in the admin editor, both following the same shape: `{ id, name, content }`, stored as their own JSON array in the configured GitHub repo (like `reviews.json`), and both use the shared `.dropdown-wrap`/`.dropdown-menu`/`.dropdown-item`/`.dropdown-remove`/`.dropdown-empty` CSS for their popup UI (each wrapper also carries its own extra class — `.blocks-dropdown-wrap`, `.tab-add-wrap` — purely so the outside-click-to-close handler in `init()` can target each independently).

- **Custom format blocks** — a "Blocks ▾" dropdown (`renderCustomBlocks`) of reusable saved *text snippets*, independent of any single review — e.g. a fully-written `[tab=Name]...[/tab]` section you want to reuse across reviews. "+ Save Block" saves the current textarea selection under a name (typed into an inline input, not `prompt()`); clicking a saved block inserts it at the cursor (`insertCustomBlock`). Stored at `data/custom-blocks.json`.
- **Saved tabs** — the review editor's "+" tab button (next to the built-in Steam Review tab and any tabs already on the review) is a dropdown, not an immediate action: its first entry is always "Blank tab" (the old default behavior, `addReviewTab()` with no args), followed by any saved tab templates (`renderTabAddDropdown`); picking one creates a new review tab pre-populated with that template's name/content (`addReviewTabFromTemplate`). "+ Save Tab" saves the *currently active tab* (name + content, via `syncActiveTabContent()` first) as a new template. Stored at `data/custom-tabs.json`.

Both features share the `fetchFile`/`pushFile`-style pattern: `fetchBlocksFile`/`pushBlocksFile` and `fetchTabsFile`/`pushTabsFile` are thin wrappers around `ghGetJson`/`ghPutJson` (see above). Both load alongside the review list on connect (`loadCustomBlocksList`/`loadCustomTabsList`, called from `loadReviewList`) — not `localStorage` — so they persist across machines/sessions. Saving/deleting either requires GitHub to be configured, same restriction as saving/deleting a review.

### Tag categories (global tag registry)

A third instance of the same save-for-reuse idea, split across two files: `data/tag-categories.json` (`[{ id, name }]`, the list of category names) and `data/tags.json` (`[{ id, name, category }]`, the global tag registry mapping every known tag name to one category, `category: ''` meaning uncategorized). `fetchTagCategoriesFile`/`fetchTagsRegistryFile` wrap `ghGetJson`; both load via `loadTagCategoriesList`/`loadTagRegistryList` from `loadReviewList`, and cache to `localStorage` (`gh_tagcats_cache`/`gh_tags_cache`) the same way `cacheReviews()` caches `gh_reviews_cache`, so the public site can read them too.

**Unlike blocks/tabs, there is no `push*` wrapper for these two files.** Every tag/category edit (add a category, rename a category, add/recategorize a tag, drag a chip onto a category line) only mutates the in-memory `tagCategories`/`tagRegistry` arrays and calls `markTagsDirty()` — no GitHub call per action, since a reorganizing session can easily touch many tags in a row. `saveReview()` only pushes `tag-categories.json`/`tags.json` when `tagsDirty` is true, bundled with the review into one `ghCommitFiles` commit (see above), then `clearTagsDirty()` and refreshes `savedTagCategories`/`savedTagRegistry` (deep-copied snapshots of the last GitHub-synced state). Any new tag/category mutation should follow this local-only + `markTagsDirty()` shape, not add its own `ghPutJson` call.

Despite `tagCategories`/`tagRegistry` being global data (not per-review), edits to them behave like a **local draft tied to whichever review is on screen**, same as every other form field — `loadReview()`/`newReview()` both call `discardTagChanges()` first, which reverts `tagCategories`/`tagRegistry` to the `savedTagCategories`/`savedTagRegistry` snapshot and clears `tagsDirty` if there were unsaved edits. This is deliberate: without it, an edit made while viewing review A and never saved would otherwise silently ride along into whatever review B you happen to save next, since the underlying data is shared.

### Unsaved-changes indicator

A plain-text "● N unsaved changes: ..." indicator (`#dirty-indicator`, next to the Save button, no dropdown — just inline text) covers **every** field, not just tags — title, summary, cover image, hours, date, recommendation, body/tab/sub-tab text, and tags/categories all count. It's driven by two independent checks, both surfaced through `updateDirtyIndicator()`: `tagsDirty` (above) and `getChangedFormFields()`, which compares the current form against `savedFormState` — a `getForm()` snapshot captured by `refreshFormSnapshot()` right after a review loads (`loadReview`/`newReview`) or saves successfully; `isFormDirty()` is just `getChangedFormFields().length > 0`.

Flat fields (title, summary, cover image, hours, date, recommendation, tags) are compared directly via `FORM_FIELD_LABELS`. Tab content (the Steam Review body plus every entry in `tabs`) goes through `diffTabContent()` instead of a flat compare, so a change is reported at the most specific level possible: a whole tab if its content changed outright (`"Tab: Pros & Cons"`), or the individual sub-tab if the tab uses `[tab=Name]` sub-tabs and only one changed (`"Tab: Pros & Cons — Combat"`, via `splitSubTabs()` from `shared.js`) — falling back to the whole-tab label if content differs but no single sub-tab explains it (e.g. a pure sub-tab reorder). Added/removed/renamed tabs and a pure tab-order change (same set of tabs, different sequence) are each reported as their own line too. `updatePreview()` — already called by essentially every field's change handler — calls `updateDirtyIndicator()` too, so no new per-field wiring is needed for a field to participate. Adding a new *flat* field to track means adding it to `getForm()`/`setForm()` and `FORM_FIELD_LABELS`; tab/sub-tab content is covered automatically since it goes through `diffTabContent()`, not the flat map.

Switching reviews (or starting a new one) without saving **discards** whatever is pending — both `discardTagChanges()` (tags/categories) and the plain overwrite `setForm`/`clearForm` already do to every other field — there's no confirmation prompt before losing it, same as it's always worked for title/summary/body. The indicator's only job is to warn you beforehand, not to gate navigation.

There's no dedicated "Manage Tags" UI — categories are managed directly in the tag chip editor (`renderTags()`), which always shows one line per known category (plus a permanent "Uncategorized" line) so every category is a visible drop target even with no tags on the current review yet. "+ Add Category" next to the Tags label uses the shared `startInlineEditor`/`cancelInlineEditor`/`onInlineEditorKeydown` helpers (see "No native browser dialogs" above); renaming a category by clicking its label is its own `renamingTagCategoryId`-tracked variant of the same shape (since, unlike "+ Add Category", more than one category's rename editor can potentially be triggered across re-renders of the same list) but still uses `onInlineEditorKeydown` for Enter/Escape. Renaming cascades to every registry tag currently filed under the old name (`tagRegistry.forEach(t => t.category === oldName ? ... )`), since a tag's category is a stored name string, not an id; skipping that cascade would silently orphan those tags back to uncategorized. Deleting a category (`startDeleteTagCategory`/`confirmDeleteTagCategory`, using the shared `deleteConfirmRow()`) cascades the same way, resetting its tags to Uncategorized rather than leaving them pointing at a name that no longer exists. Renaming or deleting an individual *tag* (as opposed to a category) happens in a separate "Manage Tags ▾" dropdown, first in the Tags label row's action bar (ahead of "+ Add Tag") (`renderManageTagsDropdown`/`toggleManageTagsDropdown`/`closeManageTagsDropdown`, same `.dropdown-wrap`/`.dropdown-menu`/`.dropdown-item`/`.dropdown-remove` shape as the Blocks/saved-tabs dropdowns) — it's the one place that lists the *entire* global registry regardless of what's on the current review, since `renderTags()`'s chip editor only ever shows tags already attached to the review being edited. Clicking a tag's name opens a `renamingTagId`-tracked inline rename editor (same input+confirm/cancel shape as the tag-category rename above); the delete `×` follows `deleteConfirmRow()` like category deletion. But unlike every other tag/category edit in this section, both **cascade into `reviews.json`**: a tag's name isn't just a registry field, it's literally the string stored in every review's `tags`/`coreTags`, so `confirmRenameRegistryTag` rewrites that string across every review (guarding against colliding with an existing different tag, case-insensitive) and `confirmDeleteRegistryTag` strips it out entirely (not just dropping its category, like category deletion does) — both push immediately via `ghCommitFiles` rather than going through the local-draft/`markTagsDirty()`/next-review-save path the rest of this section uses, since that path only ever touches whichever single review is currently open, and these need to rewrite all of them. That immediate push also flushes any other pending `tagsDirty` edits (e.g. an unsaved category rename) into the same commit, and locally updates `currentTags`/`currentCoreTags` (and patches `savedFormState`'s copies of those two fields so the dirty indicator doesn't misreport the cascade as an unsaved edit) in case the currently-open review was one of the ones affected.

**Crucially, `reviews.json`'s `tags: []` shape is unchanged** — still a flat array of tag-name strings. A tag's category is resolved by looking its name up in the registry at render/edit time, not stored per-review — this is deliberate: a tag's category is meant to be global (recategorizing "Roguelike" from Genre to Mood in the registry should apply everywhere it's used, not require editing every review that has it). `groupTagNames(tagNames, tagRegistry)` in `shared.js` does that lookup-and-group and is used by both `renderCard()`'s tag badges and the site's tag filter bar (`app.js`), as well as admin's tag chips — so a tag with no matching registry entry (e.g. a legacy tag, or one whose category/tag was since deleted) just renders as uncategorized rather than erroring; there's no cascading cleanup when a category or registry tag is deleted, consistent with the rest of this app's loose-reference style.

### Steam library import

Admin can pull the configured Steam account's owned-games list (Steam Web API key + SteamID/vanity URL in Settings) to autofill title/hours/cover art when starting a new review. Steam's API has no CORS support, so requests go through a public proxy (`corsproxy.io`, see `PROXY()` in `admin.html`) — this is a hard dependency on that third-party proxy staying up.
